# 제6장: MCP 스타일 도구 서버와 확장 아키텍처

> [!NOTE]
> 이 장의 명령 예시는 OS에 따라 문법이 다른 경우 **Windows (PowerShell)** 와 **macOS / Linux (bash·zsh)** 블록으로 나눠 표기합니다. 본인이 사용하는 OS의 블록을 따라 실행하세요. `uv run ...`처럼 양쪽이 동일한 명령은 그대로 사용하면 됩니다.

## 6.1 이 장의 목표 (Learning Objectives)

5장까지는 모델 호출, 역할 분리, Streamlit UI를 하나의 애플리케이션 안에서 실행했습니다. 6장에서는 그 기능을 HTTP endpoint로 분리해, 다른 클라이언트나 에이전트가 같은 기능을 도구처럼 호출할 수 있는 구조를 확인합니다.

이 저장소의 6장은 포털에 MCP 서버를 등록하는 완성 예제가 아니라, MCP/원격 도구 연결을 이해하기 위한 로컬 최소 구현입니다. 실습은 두 흐름만 남겼습니다.

- **세션 분리 이해:** 대화 기록을 클라이언트가 매번 들고 다니지 않고 서버가 세션 ID 기준으로 관리하는 구조를 확인합니다.
- **도구 서버 이해:** RAG 기능을 애플리케이션 내부 함수가 아니라 독립 HTTP endpoint로 노출하는 방식을 확인합니다.
- **MCP 연결 감각 익히기:** Foundry 포털의 원격 MCP/로컬 MCP 도구가 어떤 endpoint와 payload 개념 위에 서 있는지 로컬 코드로 먼저 이해합니다.

## 6.2 현재 Foundry 도구 화면과의 연결

현재 Foundry 포털의 `도구` 화면에는 `원격 MCP`, `로컬 MCP`, `사용자 지정`, 그리고 Foundry MCP Server, GitHub, Azure Cosmos DB 같은 MCP 카탈로그 항목이 보일 수 있습니다. 이 장의 MCP 실습은 원격 도구 연결 개념을 이해하기 위한 로컬 실행 실습으로 사용합니다.

이 저장소의 6장 코드는 포털 카탈로그에서 MCP 서버를 바로 등록하는 예제가 아니라, 로컬에서 MCP 스타일 HTTP 도구 서버를 직접 만들고 클라이언트가 호출하는 예제입니다. 즉, 포털에서 보이는 `원격 MCP` 도구를 이해하기 위한 최소 구현을 먼저 손으로 실행해 보는 단계입니다.

## 6.3 왜 MCP 스타일 서버가 필요한가?

5장에서 만든 multi-agent 흐름은 한 Python 프로세스 안에서 이전 결과를 다음 prompt로 넘깁니다. 운영 환경에서는 이 기능을 여러 앱, 작업자, 도구 서버가 함께 써야 하므로 요청/응답 형식, 세션 ID, endpoint 경계가 필요합니다.

6장의 목표는 MCP 표준 전체를 구현하는 것이 아니라, “에이전트 기능을 도구 서버로 분리하면 어떤 모양이 되는가?”를 손으로 확인하는 것입니다.

## 6.4 핵심 개념

이 장에서는 네 가지 관점만 집중해서 봅니다.

1. **세션:** 대화나 작업의 경계입니다. 6.4 서버는 `session_id`로 대화 기록을 구분합니다.
2. **서비스:** 모델 호출이나 RAG 검색처럼 독립적으로 호출할 수 있는 기능입니다.
3. **메시지:** 클라이언트와 서버가 주고받는 JSON 요청/응답입니다.
4. **컨텍스트:** 서버가 보관하거나 RAG 검색으로 찾아 모델 입력에 넣는 추가 정보입니다.

## 6.5 선택 실습: 세션을 가진 채팅 서버

`6.4.2_mcp_flask_server.py`와 `6.4.3_mcp_client.py`는 세션과 대화 컨텍스트를 관리하고 Foundry 모델 호출을 감싸는 Flask 기반 채팅 서버 예제입니다.

서버와 클라이언트는 터미널을 나눠 실행합니다.

```bash
uv run Chapter6_MCP_Protocol/6.4.2_mcp_flask_server.py
uv run Chapter6_MCP_Protocol/6.4.3_mcp_client.py
```

이 예제는 `/session`으로 세션을 만들고 `/chat`으로 세션별 대화 기록을 유지합니다. 세션 상태는 서버 프로세스의 메모리에만 저장되므로 서버를 종료하면 사라집니다.

서버 터미널에는 `[새 세션 생성]`, `[클라이언트 요청]`, `[서버 응답]`이 출력됩니다. 따라서 클라이언트가 어떤 질문을 보냈고 서버가 어떤 모델 답변을 반환했는지 서버 쪽에서도 바로 확인할 수 있습니다.

## 6.6 필수 실습: RAG 도구 서버

`6.6.1_mcp_server_ai_search.py`와 `6.6.2_mcp_client_ai_search.py`는 회사 정책 RAG를 독립 도구 서버처럼 노출합니다. 4장에서 만든 RAG 기능이 6장에서는 외부 도구 프로토콜의 서비스로 바뀌는 흐름을 확인합니다.

```bash
uv run Chapter6_MCP_Protocol/6.6.1_mcp_server_ai_search.py
uv run Chapter6_MCP_Protocol/6.6.2_mcp_client_ai_search.py
```

6시간 본 실습에서는 이 RAG 도구 서버를 6장의 대표 실행으로 권장합니다. 시간이 부족하면 6.4 세션 서버는 코드 설명만 하고 넘어가도 됩니다.

RAG 도구 서버도 서버 터미널에 `[클라이언트 요청]`과 `[서버 응답]`을 출력합니다. 클라이언트 터미널의 `도구 응답`과 서버 터미널의 `답변`이 같은지 비교하면 HTTP 도구 서버가 요청을 받아 RAG 답변을 돌려주는 흐름을 쉽게 확인할 수 있습니다.

## 6.7 실행 결과를 볼 때 확인할 점

6장 예제는 서버와 클라이언트를 터미널 두 개에서 나눠 실행합니다. 서버 터미널에서는 endpoint가 요청을 받는지 보고, 클라이언트 터미널에서는 어떤 JSON 요청과 응답이 오가는지 확인합니다.

각 Python 파일은 실행 시작 시 학습 목표 안내 블록을 먼저 출력합니다. 서버는 계속 실행되는 프로세스이므로, 안내 블록 이후에는 새 요청이 들어올 때마다 질문과 답변 로그가 추가되는지 관찰합니다.

6.4의 세션 상태는 예제 서버 프로세스의 메모리 안에 저장됩니다. 서버를 종료하면 세션 기록은 사라집니다. 6.6은 RAG 기능을 HTTP 도구 endpoint로 감싼 구조를 보여주며, 이 endpoint가 나중에 원격 MCP 도구로 등록될 수 있는 최소 단위입니다.
