# 7장 설명: Human Review, Reflection, Guardrails

## 이 장의 목적

7장은 “답변이 나온다”를 넘어 안전하게 운영하는 방법을 다룬다. 중요한 질문은 모델이 얼마나 똑똑한지가 아니라 언제 멈추고, 누가 승인하며, 무엇을 기록하는가다.

## 핵심 파일

| 파일 | 역할 |
| --- | --- |
| `7.3.1_langgraph_interrupt.py` | 중간에 사람 피드백을 받는 흐름 |
| `7.4.1_reflection_loop.py` | 초안 후 비평·개선 호출 |
| `7.6.1_guardrails.py` | 차단·사람 검토·허용 라우팅 |

## Human-in-the-Loop

AI가 확신이 없거나 실패 비용이 큰 경우 자동 실행을 멈추고 사람에게 넘긴다. 금액 승인, 개인정보, 법률·인사 판단, 외부 시스템 변경에 특히 중요하다.

현재 `7.3.1_langgraph_interrupt.py`는 파일명과 달리 완전한 영구 중단/재개 workflow가 아니라 로컬 입력으로 피드백을 받는 교육용 흐름이다.

## Reflection

```text
초안 생성 -> 초안을 다시 입력 -> 비평/개선 -> 최종본
```

Reflection은 모델 내부의 숨겨진 생각을 읽는 기능이 아니다. 이전 출력을 다음 호출에 넣는 외부 loop다. 품질이 좋아질 수 있지만 호출 횟수와 비용이 늘고 같은 모델이 오류를 놓칠 수도 있다.

## Guardrail 흐름

`7.6.1_guardrails.py`는 다음 단계를 보여준다.

1. 명시적 prompt injection 패턴 검사
2. 민감 정보와 정책 질문 규칙 검사
3. 필요한 경우 모델 분류
4. `BLOCK`, `NEEDS_HUMAN_REVIEW`, `ALLOW` 결정
5. 허용된 질문만 정책 context로 답변
6. 답변이 context에 근거했는지 재검사

규칙은 빠르고 예측 가능하지만 우회 표현을 놓칠 수 있다. 모델 분류는 유연하지만 오판 가능성과 비용이 있다. 실제 서비스는 두 방식을 조합한다.

## Guardrail은 여러 층이어야 한다

- 인증과 RBAC
- 입력 검증
- 문서 접근 권한 필터
- 허용 도구 목록
- 민감 정보 마스킹
- 사람 승인
- 출력 근거성 검사
- rate limit과 감사 기록

Prompt에 “안전하게 행동하라”고 쓰는 것만으로는 충분하지 않다.

## 운영 관점

서비스에서는 정답성뿐 아니라 차단 오탐률, 위험 요청 누락률, 근거성, 응답 시간, 요청당 비용, 도구 실패율을 측정해야 한다.

## 이 장을 읽고 답할 수 있어야 하는 질문

1. 어떤 업무에서 사람 승인이 필요한가?
2. Reflection이 내부 사고 공개가 아닌 이유는 무엇인가?
3. 규칙 검사와 모델 분류를 함께 쓰는 이유는 무엇인가?
4. Guardrail을 prompt 하나로 구현하면 왜 부족한가?
5. 안전장치의 오탐과 누락을 어떻게 평가할 수 있는가?
