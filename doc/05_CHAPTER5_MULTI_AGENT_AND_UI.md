# 5장 설명: Multi-agent 협업과 Streamlit UI

## 이 장의 목적

5장은 복잡한 작업을 역할별 모델 호출로 나누고, 결과를 사용자가 조작할 수 있는 웹 화면으로 감싼다.

## Multi-agent의 실제 모습

`5.2.1_multi_agent_system.py`는 연구원 역할의 결과 문자열을 작가 역할의 입력에 넣는다.

```text
주제 -> 연구원 호출 -> 연구 노트 -> 작가 호출 -> 블로그 초안
```

이 예제에서 독립 agent 서버끼리 직접 대화하는 것은 아니다. Python 코드가 호출 순서를 정하고 이전 결과를 다음 prompt로 전달한다. 즉 역할 기반 sequential workflow다.

## 역할을 나누는 이유

- 각 단계의 책임과 prompt가 명확해진다.
- 중간 결과를 검사할 수 있다.
- 특정 단계만 재시도하거나 다른 모델로 교체할 수 있다.
- 단계별 품질 평가가 가능하다.

Agent 수를 늘리면 호출 비용, 지연 시간, 오류 전파도 증가한다. 한 번의 호출로 충분한 작업을 억지로 분할할 필요는 없다.

## 핵심 파일

| 파일 | 역할 |
| --- | --- |
| `5.2.1_multi_agent_system.py` | 연구원과 작가 역할 순차 연결 |
| `5.3.1_streamlit_app.py` | 단일 질문 웹 UI |
| `5.3.2_streamlit_chatbot_with_memory.py` | session state 기반 대화 UI |

## Streamlit의 역할

Streamlit은 Python 코드로 빠르게 웹 UI를 만드는 도구다. 사용자는 브라우저에서 질문하고 결과를 볼 수 있으며, 모델 호출은 서버 측 Python 프로세스에서 수행된다.

Memory 예제는 `st.session_state.messages`에 대화를 저장한다. 브라우저 세션이 유지되는 동안만 기억하는 것처럼 보이며 앱 재시작이나 새 세션에서는 초기화된다.

## UI가 생겨도 아직 부족한 것

- 사용자 인증과 권한
- 사용자별 상태 격리
- 입력 크기와 호출 횟수 제한
- secret 관리
- 오류 화면과 timeout
- 배포, 확장, 장애 대응
- 개인정보 보존과 삭제 정책

## 이 장을 읽고 답할 수 있어야 하는 질문

1. 5장의 multi-agent는 결과를 어떻게 전달하는가?
2. 역할 분리의 장점과 비용은 무엇인가?
3. Streamlit session state는 영구 저장소인가?
4. 웹 화면을 붙이는 것과 프로덕션 서비스의 차이는 무엇인가?
5. 어떤 작업은 multi-agent로 나누지 않는 편이 좋은가?
