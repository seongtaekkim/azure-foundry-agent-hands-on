# 제4장: 핵심 AI 에이전트 디자인 패턴: 진정으로 유능한 에이전트 구축하기

## 4.1 이 장의 목표 (Learning Objectives)

3장에서 우리는 API key 기반 prompt-style agent를 만들고, 외부 정보가 필요한 질문에서 도구 연결의 필요성을 확인했습니다. 이제 현실 세계의 문제를 해결하기 위해 로컬 함수, 문서 검색, Azure AI Search로 이어지는 도구 패턴을 단계적으로 붙입니다.

이 장에서는 수십 년간의 소프트웨어 공학에서 증명된 '디자인 패턴'의 개념을 AI 에이전트 개발에 적용합니다. 디자인 패턴은 반복적으로 발생하는 문제에 대한 검증된 해결책의 청사진입니다. 이 장을 마치면, 여러분은 더 이상 단편적인 기능을 추가하는 것을 넘어, **검증된 설계 방법론을 바탕으로 안정적이고 확장 가능한 전문가 수준의 AI 에이전트를 구축**하는 핵심 역량을 갖추게 될 것입니다.

- **다중 도구 활용 능력:** 에이전트에게 여러 개의 도구를 제공하고, 주어진 문제에 가장 적합한 도구를 **스스로 선택하여 사용**하게 만드는 'Tool Use' 패턴을 마스터합니다.
- **전문 지식 내재화:** LLM이 알지 못하는 **기업 내부 문서나 특정 도메인 지식**을 에이전트가 학습하고 참조하게 만드는, 현존하는 가장 중요한 산업 패턴인 **RAG(Retrieval Augmented Generation)**를 직접 구현할 수 있습니다.
- **고급 문서 이해:** Document Intelligence나 Foundry 문서 이해 모델의 추출 결과를 RAG 파이프라인에 통합하는 확장 방식을 이해합니다.
- **프로덕션급 RAG 확장 설계:** 로컬 RAG 원리를 이해한 뒤, 대규모 문서에서는 **Azure AI Search** 같은 관리형 검색 서비스로 확장해야 하는 이유를 설명할 수 있습니다.

## 4.2 디자인 패턴 1: 도구 사용 (Tool Use Pattern) 심화

3장에서 만든 에이전트는 '망치' 하나만 가진 목수와 같습니다. 이제 우리는 에이전트에게 망치, 톱, 드라이버가 모두 들어있는 **'공구함'**을 주고, 상황에 맞는 연장을 스스로 꺼내 쓰게 만들어야 합니다.

> **[Key Concept]**
> 진정한 도구 사용 패턴의 핵심은 단순히 도구를 '호출'하는 것이 아니라, 여러 도구 중에서 주어진 과업에 가장 적합한 도구를 **'선택(Selection)'**하고, 그 도구의 결과를 **'해석(Interpretation)'**하여 다음 행동을 결정하는 **자율적인 의사결정 과정**에 있습니다.

### 4.2.1 Hands-on: 멀티-툴(Multi-Tool) 에이전트 구축하기

이 실습은 `4.2.1_multi_tool_agent.py` 파일에 해당합니다. 실행하면 `[학습 목표]`, `[사용자 요청]`, `[사용 가능한 도구 후보]`, `[로컬 도구 실행 결과]`, `[모델이 도구 결과를 종합한 최종 답변]` 순서로 출력됩니다. 교육용으로 계산기와 정책 조회 결과를 먼저 로컬 함수로 만들고, Foundry 모델이 어떤 결과를 어떤 사용자 요구에 연결해야 하는지 설명하게 합니다. 실제 서비스에서는 이 로컬 함수를 Foundry function tool 또는 외부 MCP 도구로 승격할 수 있습니다.

## 4.3 디자인 패턴 2: RAG (Retrieval Augmented Generation)

LLM은 세상의 거의 모든 공개 지식을 알고 있지만, **당신 회사의 내부 정책 문서나, 어제 막 새로 출시된 제품의 매뉴얼**에 대해서는 아무것도 알지 못합니다. RAG는 이 문제를 해결하는 가장 강력하고 표준적인 디자인 패턴입니다.

> **[Key Concept]**
> RAG는 LLM에게 **'오픈북 시험(Open-book Exam)'**을 치르게 하는 기술입니다. LLM이 모르는 질문을 받으면, 정답이 적힌 **'참고 자료(우리의 문서)'**를 실시간으로 찾아서 함께 제공하여, LLM이 그 자료를 기반으로 **'사실에 근거한(Fact-based)'** 답변을 생성하게 만듭니다. 이를 통해 LLM의 가장 큰 약점인 **'환각(Hallucination, 거짓말 현상)'**을 획기적으로 줄일 수 있습니다.

### 4.3.1 RAG의 두 단계: 색인(Indexing)과 검색/생성(Retrieval/Generation)

- **준비 단계: 색인 (Indexing):** 우리의 문서를 '지식 창고(Vector Store)'에 미리 저장해두는 과정입니다. (Load -> Split -> Embed -> Store)
- **실행 단계: 검색 및 생성 (Retrieval & Generation):** 사용자가 질문을 하면 실시간으로 일어나는 과정입니다. (Query -> Retrieve -> Augment -> Generate)

### 4.3.2 Hands-on: 우리 회사 정책을 답변하는 RAG 에이전트 구축하기

이 실습은 `4.3.2_rag_agent.py` 파일과 `company_policy.txt` 파일을 사용합니다. 에이전트가 우리만의 내부 문서를 기반으로 정확한 답변을 생성하는 과정을 구현합니다.

### 4.3.3 문서 이해 결과를 활용한 고급 RAG

단순 텍스트 파일을 넘어 PDF나 이미지에 담긴 복잡한 표, 레이아웃, 다국어 텍스트를 이해하는 것은 RAG의 중요한 과제입니다. 이 실습은 `4.3.3_advanced_rag_mistral.py` 파일에 해당하며, 교육 환경에서는 `company_policy.pdf`를 `pypdf`로 읽어 추출한 텍스트를 `extracted_document` 딕셔너리로 구성합니다.

`4.3.2_rag_agent.py`가 원문 텍스트를 청크로 나누고 embedding 검색으로 관련 문맥을 찾는 기본 RAG라면, `4.3.3_advanced_rag_mistral.py`는 PDF 파일에서 추출한 텍스트와 문서 구조 힌트를 context로 넣어 답하게 합니다. 즉 4.3.3의 핵심은 검색 알고리즘이 아니라 “PDF/OCR/문서 이해 결과를 LLM이 읽기 좋은 구조화 context로 바꾸는 방법”입니다.

### 4.4 Hands-on: 프로덕션급 RAG 확장 설계

수십만 페이지의 문서를 다루는 엔터프라이즈 환경에서는 로컬 검색의 한계를 넘어, **Azure AI Search**와 같은 프로덕션급 검색 엔진이 필요합니다. 이 장의 `4.4.4.2_advanced_rag_index.py`, `4.4.4.3_advanced_rag_agent.py`, `4.4.4.5_advanced_rag_multiQueryRetriever.py`, `4.4.4.6_multi_tool_agent.py`는 실제 Search 리소스 구축이 아니라 Foundry embedding, 쿼리 재작성, RAG 답변, 도구 라우팅 패턴을 단계적으로 보여줍니다.

`4.4.4.6_multi_tool_agent.py`는 단순 키워드 if문으로 도구를 고르는 예제가 아니라, 모델이 먼저 사용 가능한 도구 목록을 보고 JSON 형태의 tool plan을 생성하는 agentic router 예제입니다. 이후 Python 코드는 AI가 선택한 계획에 따라 실제 로컬 도구를 실행합니다. 현재 예제의 도구 후보는 RAG search, calculator, human review, specialist handoff plan이며, 휴가 규정 요청에서는 회사 정책 문서 RAG 검색과 HR 전문가 위임 계획이 실제로 실행됩니다.

실행 흐름은 다음과 같습니다.

```text
사용자 요청
    -> AI planner가 필요한 도구 판단
    -> JSON tool plan 출력
    -> route_tools()가 계획에 따라 로컬 도구 실행
    -> RAG/specialist handoff plan/human review/calculator 결과 수집
    -> 최종 모델이 실행 결과를 종합해 답변
```

## 4.5 실행 결과를 볼 때 확인할 점

모든 4장 실습 파일은 시작 시 학습 목표 안내 블록을 출력합니다. 먼저 어떤 패턴을 보는 파일인지 확인하고, 이어서 로컬 도구 결과, 검색된 context, 모델에 전달되는 prompt가 어떻게 최종 답변으로 이어지는지 봅니다.

RAG 예제는 모델이 모든 것을 기억해서 답하는 구조가 아닙니다. 먼저 로컬 문서나 인덱스에서 관련 문맥을 찾고, 그 검색 결과를 prompt에 포함한 뒤 모델에게 답변을 요청합니다. 따라서 터미널 출력에서는 “검색된 문맥이 무엇인지”와 “모델 응답이 그 문맥을 근거로 삼는지”를 함께 확인합니다.

`4.3.2_rag_agent.py`와 `4.3.3_advanced_rag_mistral.py`는 의도적으로 다른 흐름입니다. 4.3.2는 원문 텍스트 검색 기반 RAG이고, 4.3.3은 PDF에서 추출한 텍스트를 문서 구조 context로 바꾸는 문서 이해형 RAG입니다.

`4.4.4.5_advanced_rag_multiQueryRetriever.py`는 사용자 질문을 여러 검색 질의로 확장합니다. 이때 추가 질의는 별도 지식 저장소가 아니라 검색 품질을 높이기 위한 재작성 결과이며, 최종 답변은 검색된 문맥을 다시 모델 입력으로 넣어 생성됩니다.

`4.4.4.6_multi_tool_agent.py`는 먼저 모델에게 도구 선택 계획을 만들게 하므로, 출력에서 `[AI가 판단한 도구 사용 계획]`을 확인합니다. 이어서 `[라우터가 선택하고 실행한 도구]`에서 실제로 어떤 도구가 실행되었는지, RAG 검색 결과가 최종 답변에 어떻게 반영되는지 확인합니다. 이 예제는 API key 기반 실습이므로 specialist handoff plan은 실제 원격 호출이 아니라 prompt 단계 분리나 MCP 도구 서버로 확장할 수 있는 위임 계획으로 출력됩니다.
