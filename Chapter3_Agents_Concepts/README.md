# 제3장: AI 에이전트의 세계: 기본 개념과 프레임워크

## 3.1 이 장의 목표 (Learning Objectives)

지금까지 우리는 AI에게 질문하고 응답을 받는 '대화'를 나누었고(1장), AI의 도움을 받아 코드를 더 빠르고 효율적으로 작성하는 '협업'을 경험했습니다. 이제 우리는 AI 개발의 다음 차원으로 도약합니다. 바로 AI에게 **목표를 부여하고, 스스로 생각하고, 행동하게 만드는 '위임'의 단계**입니다. 이것이 바로 '지능형 AI 에이전트'의 세계입니다.

이 장을 마치면, 여러분은 다음과 같은 핵심적인 역량을 갖추게 될 것입니다.

- **개념 정립:** AI 에이전트를 '생각하고 행동하는 프로그램'으로 정의하고, 그 핵심 구성 요소인 **Sense-Think-Act** 사이클을 명확하게 설명할 수 있습니다.
- **Agent 실행 구조 이해:** 왜 단순 LLM 호출을 넘어 역할 지시문, 상태, 도구 연결 지점이 필요한지 이해합니다.
- **기본 에이전트 구축:** Python과 API key 기반 Responses API를 사용하여 첫 번째 prompt-style agent를 직접 구축할 수 있습니다.
- **동작 원리 분석:** 에이전트가 어떤 '생각'의 과정을 거쳐 행동을 결정하는지, 그 내부적인 '추론 과정(Chain of Thought)'을 관찰하고 분석할 수 있습니다.
- **미래 트렌드 파악:** 2025년 이후 AI 에이전트 개발을 주도할 핵심 트렌드(상태 관리, 다중 에이전트 협업 등)를 이해하고 다음 학습 단계를 준비합니다.

## 3.2 에이전트란 무엇인가?: 챗봇을 넘어서

가장 먼저, '챗봇'과 '에이전트'의 근본적인 차이를 이해해야 합니다.

- **챗봇 (Chatbot):** 주어진 질문에 대해 학습된 지식 내에서 **응답(React)**하는 시스템입니다.
- **에이전트 (Agent):** 주어진 목표를 달성하기 위해, 환경을 **인식(Sense)**하고, 계획을 수립하여 **생각(Think)**한 뒤, 외부 세계에 영향을 미치는 **행동(Act)**을 수행하는 시스템입니다.

> **[Mental Model Shift]**
> **챗봇이 '지식인'이라면, 에이전트는 '행동하는 비서'입니다.** "오늘 날씨 어때?"라고 물었을 때, 챗봇은 어제 학습한 데이터를 기반으로 "일반적으로 맑습니다"라고 대답할 수 있습니다. 하지만 에이전트는 "현재 서울의 날씨를 알아봐야겠다"고 생각하고, 실시간 날씨 API라는 도구를 사용하여 "현재 서울의 기온은 25도이며, 맑습니다"라는 정확한 정보를 가져와 행동의 결과물을 보여줍니다.

이 모든 과정은 **Sense-Think-Act** 라는 핵심적인 순환 고리를 통해 이루어집니다.

## 3.3 왜 프레임워크가 필요한가?: 똑똑한 뇌(LLM)에 강력한 몸체(Framework)를

LLM 호출과 `if/else` 문만으로 에이전트를 만들 수 있지 않을까요? 가능하지만 곧 거대한 장벽에 부딪히게 됩니다. AI 에이전트 프레임워크는 바로 이 장벽을 해결하기 위해 등장했습니다.

- **상태 관리 (State Management):** 여러 단계를 거치는 작업의 이전 결과를 기억하고 활용하게 해줍니다.
- **도구 통합 (Tool Integration):** 수많은 외부 도구들을 일관된 방식으로 에이전트에게 연결해줍니다.
- **추론 사이클 (Reasoning Cycle):** '이 도구를 썼는데 실패했네, 그럼 다른 도구를 써볼까?'와 같이 LLM이 스스로 생각하고 행동을 수정하는 복잡한 루프를 쉽게 구현하도록 돕습니다.
- **보일러플레이트 코드 제거:** 반복적인 코드를 대신 처리해주어 핵심 로직에만 집중하게 합니다.

이 커리큘럼에서는 API key 기반 Responses API에 역할 지시문을 부여해 에이전트형 실행 흐름을 구축합니다.

`3.4.2_agent_app.py` 파일의 코드를 실행하고, 에이전트의 동작을 관찰합니다.

### 3.4.3 에이전트 실행 및 사고 과정 관찰

코드를 실행한 뒤, 에이전트에게 모델 지식만으로는 확정하기 어려운 **실시간 정보**에 대해 질문해 보세요. 예를 들어:

> **당신:** 현재 과천 날씨를 알려줘. 실시간 도구가 없다면 어떤 연결이 필요한지도 설명해줘.

터미널에 먼저 출력되는 학습 목표 안내 블록을 읽고, 이어서 agent 이름, 모델에 전달된 prompt, 응답, trace span 흐름을 관찰하세요. 이후 4장에서 로컬 도구와 RAG를 붙이고, 5장에서 다중 에이전트 협업을 확인합니다. 8장에서는 실제 Foundry Agent Service 실행으로 확장하며 **Sense-Think-Act** 사이클이 운영형 agent 구조로 어떻게 이어지는지 확인합니다.

### 3.4.4 선택 실습: 계산기 도구가 필요한 이유

`3.4.4_agent_with_calculator.py`는 계산 요청을 예로 들어, 모델 지식만으로 처리하기보다 명시적인 도구가 필요한 상황을 설명합니다. 정식 도구 패턴은 4장에서 더 자세히 다룹니다.

### 3.4.5 상태 유지 에이전트

`3.4.5_stateful_agent.py`는 대화 기록을 유지하면서 같은 공통 실행 기반을 재사용하는 예제입니다. 이 파일은 `3.4.2`와 함께 3장에서 권장하는 실행 대상입니다(루트 README의 "권장 실행 순서" 기준). 반면 바로 위 `3.4.4` 계산기 예제는 도구의 필요성을 보여주는 선택 실습입니다.

이 예제의 상태는 Foundry 서버에 저장되는 영구 메모리가 아니라, 실행 중인 Python 프로세스의 `messages` 리스트에 저장됩니다. `run_threaded_prompt()`가 사용자 메시지와 assistant 응답을 같은 리스트에 append하고, 다음 호출 때 누적된 리스트 전체를 다시 모델에 보냅니다. 프로그램을 종료하면 이 대화 상태는 사라집니다.

1턴 응답 뒤에는 Enter 입력을 기다리도록 구성되어 있습니다. 터미널에서 1턴 응답을 확인한 뒤 Enter를 누르면, 2턴 질문이 이전 대화 기록과 함께 전달되는 흐름을 관찰합니다.
